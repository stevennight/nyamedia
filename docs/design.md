# Emby Multi-Source Direct Play Gateway Design

## 1. Goal

构建一个独立服务，为 Emby 提供统一的多数据源媒体入口，支持：

- 115 网盘
- 123pan
- OpenList / AList
- 本地目录或其他后续数据源

该服务本身不负责刮削、元数据管理和播放器能力，只负责：

- 统一目录抽象
- 生成 Emby 可识别的 `.strm` 文件
- 提供统一播放入口
- 在直链跳转和代理转发之间做兼容处理

目标是让 Emby 只依赖本服务，而不是直接耦合某个网盘或挂载工具。

## 2. Non-Goals

本项目不负责以下内容：

- 替代 Emby / Jellyfin 的媒体库功能
- 自建刮削器或 TMDB 元数据系统
- 自建播放器
- 完整文件系统挂载实现
- 网盘资源上传、离线下载、转存等运营能力

## 3. Core Principles

- 优先使用最小可行方案，首版采用 `STRM + HTTP Gateway`
- 所有数据源通过统一 `Provider` 接口接入
- 播放链路默认走 `302` 直链，兼容性问题时回退到代理模式
- 尽量把状态集中在服务端，避免 Emby 侧维护复杂配置
- 保持对后续数据源扩展友好，不引入平台专有逻辑到核心层

## 5. Main Modules

### 5.1 Provider Layer

每个数据源实现同一组能力：

- 列目录
- 查询文件信息
- 获取文件可播放地址
- 可选：增量变更探测

统一接口建议：

```go
type Provider interface {
    ID() string
    Type() string
    List(ctx context.Context, path string) ([]Entry, error)
    Stat(ctx context.Context, path string) (*Entry, error)
    GetDirectLink(ctx context.Context, path string) (*DirectLinkResult, error)
}
```

数据结构：

```go
type Entry struct {
    ID       string
    Name     string
    Path     string
    IsDir    bool
    Size     int64
    ModTime  time.Time
    MimeType string
}

type DirectLinkResult struct {
    URL           string
    Headers       map[string]string
    ExpireAt      *time.Time
    SupportsRange bool
}
```

### 5.2 Catalog Layer

负责把多个 Provider 暴露成统一目录树。

职责：

- 维护逻辑媒体根目录到 provider 根目录的映射
- 统一路径规范
- 过滤非媒体文件
- 为索引器提供标准化遍历结果

示例逻辑映射：

```yaml
libraries:
  - id: movies
    name: Movies
    mounts:
      - provider: pan115
        source_path: /影视/电影
        target_path: /Movies/115
      - provider: pan123
        source_path: /movie
        target_path: /Movies/123pan
```

### 5.3 Indexer Layer

负责扫描 Catalog 并生成 `.strm` 文件。

职责：

- 全量扫描
- 增量扫描
- 根据规则生成目标目录结构
- 输出 `.strm` 文件内容
- 可选复制同名字幕、nfo 占位或辅助文件

STRM 文件内容格式：

```text
http://gateway-host:7001/stream/{provider_id}/{escaped_path}
```

可选增加模式参数：

```text
http://gateway-host:7001/stream/{provider_id}/{escaped_path}?mode=redirect
```

### 5.4 Gateway Layer

负责接收 Emby 播放请求并转换为真正的播放地址。

核心路由：

- `GET /stream/{provider}/{path...}`
- `HEAD /stream/{provider}/{path...}`

支持两种工作模式：

1. `redirect`
返回 `302` 到真实直链

2. `proxy`
服务端拉取真实资源并转发给客户端

模式选择策略：

- 默认 `redirect`
- 明确传 `mode=proxy` 时强制代理
- 可按客户端类型配置默认策略
- 某些 provider 不支持稳定 302 时可强制代理

### 5.5 Cache Layer

用于缓存短期状态，减少频繁请求源站。

建议缓存项：

- 目录列表缓存
- 文件元数据缓存
- 直链缓存
- 扫描状态缓存

直链缓存需要特别注意：

- 只缓存短时有效链接
- 缓存 TTL 不得超过源链接实际有效期
- 链接即将过期时优先刷新

### 5.6 Admin Layer

首版只需要最简单能力：

- 查看 provider 状态
- 手动触发扫描
- 查看最近扫描日志
- 查看最近播放请求日志
- 配置 redirect/proxy 策略

首版可先用 API，不强制要求完整前端页面。

## 6. Playback Flow

### 6.1 Emby Scan Flow

```text
Provider -> Catalog -> Indexer -> generate .strm -> Emby scan library
```

### 6.2 Redirect Play Flow

```text
Emby client
  -> request /stream/pan115/movie/test.mkv
  -> gateway resolves provider
  -> provider returns temporary direct link
  -> gateway responds 302 Location: real_url
  -> client requests real_url directly
```

### 6.3 Proxy Play Flow

```text
Emby client
  -> request /stream/pan115/movie/test.mkv?mode=proxy
  -> gateway resolves provider
  -> provider returns temporary direct link
  -> gateway requests upstream with Range/header passthrough
  -> gateway streams response back to client
```

## 7. API Design

### 7.1 Public Playback API

#### `GET /stream/{provider}/{path...}`

Query 参数：

- `mode=redirect|proxy`
- `token=...` 可选，用于外部访问保护

行为：

- 校验 provider 是否存在
- 解析文件路径
- 查找或刷新直链缓存
- 根据策略返回 302 或代理流

#### `HEAD /stream/{provider}/{path...}`

用于部分客户端的探测请求。

要求：

- 在 `redirect` 模式下可直接返回重定向
- 在 `proxy` 模式下尽量返回上游文件头信息

### 7.2 Admin API

#### `GET /api/providers`

返回已注册 provider 列表和状态。

#### `POST /api/scan/full`

触发全量扫描。

#### `POST /api/scan/library/{id}`

触发单库扫描。

#### `GET /api/tasks`

查看当前和历史任务。

#### `GET /api/logs/playback`

查看播放请求日志。

## 8. Config Design

配置存储采用“数据库优先”方案：

- 业务配置主存储使用 SQLite
- 启动引导配置只保留少量基础项
- YAML 或 JSON 仅用于导入导出和初始化引导

这样划分的原因：

- 更适合 Web 管理界面做增删改查
- 更容易做字段校验、唯一约束和版本迁移
- 任务、日志、缓存等运行时数据天然适合数据库
- 避免用户误改配置文件导致服务无法启动

### 8.1 Bootstrap Config

启动配置文件只负责应用启动所需的最小参数。

建议保留：

- 服务监听地址
- 数据目录位置
- SQLite 数据库路径
- 初始管理员账号或密码
- 日志等级

示例：

```yaml
server:
  host: 0.0.0.0
  port: 7001
  public_base_url: http://127.0.0.1:7001

storage:
  data_dir: ./data
  db_path: ./data/app.db
  strm_output_dir: ./data/strm

auth:
  bootstrap_username: admin
  bootstrap_password: change-me

logging:
  level: info
```

### 8.2 Database As Source Of Truth

以下配置项不再作为主配置文件内容，而是由数据库维护：

- providers
- libraries
- mounts
- playback settings
- user agent rules
- admin settings
- tokens and secrets

应用启动流程建议为：

1. 读取 bootstrap 配置
2. 初始化数据库连接
3. 执行 schema migration
4. 从数据库加载运行配置
5. 启动 HTTP 服务和后台任务

### 8.3 Import And Export

为了兼顾迁移和备份，系统仍应支持导入导出：

- 导出当前数据库配置为 YAML 或 JSON
- 从 YAML 或 JSON 导入到数据库
- 导入时先做结构校验，再执行覆盖或合并策略

建议规则：

- 默认导出时脱敏敏感字段
- 导入敏感字段时支持显式覆盖
- 导入完成后返回变更摘要

### 8.4 Sensitive Config Handling

敏感数据建议单独对待：

- cookie
- access token
- provider secret
- admin password hash

原则：

- 数据库存储时至少做字段级脱敏展示
- 管理 API 不返回明文敏感值
- 日志中不得打印完整凭据
- 如条件允许，可对敏感字段做应用层加密

## 10. STRM Generation Rules

### 10.1 File Selection

仅为以下媒体文件生成 `.strm`：

- `.mkv`
- `.mp4`
- `.avi`
- `.ts`
- `.m2ts`
- `.mov`
- `.wmv`

字幕文件可作为后续增强：

- `.srt`
- `.ass`
- `.ssa`
- `.sub`

### 10.2 Output Path Rule

源路径：

```text
provider pan115: /影视/电影/Avatar (2009)/Avatar.2009.mkv
```

目标路径：

```text
data/strm/Movies/115/Avatar (2009)/Avatar.2009.strm
```

### 10.3 Conflict Strategy

当多个 provider 下出现同名文件：

- 默认保留不同根目录隔离
- 不在首版自动去重合并
- 版本合并交给 Emby 多版本电影能力处理

## 11. Provider Implementation Notes

### 11.1 115 Provider

职责：

- 登录态维护
- 路径到文件 ID 的映射
- 获取临时下载或播放链接

风险点：

- cookie 过期
- 风控限制
- 同时播放限制
- 链接有效期短

### 11.2 123pan Provider

职责与 115 类似，但需单独封装鉴权和直链接口。

风险点：

- API 稳定性
- 分享链与正式下载链差异
- Header 或签名要求

### 11.3 OpenList Provider

实现方式更简单：

- 用 OpenList API 列目录
- 用其现有直链能力获取最终 URL

这类 provider 可以作为“适配器 provider”，帮助快速兼容更多后端。

## 12. Compatibility Strategy

### 12.1 Why Redirect First

优点：

- 服务器几乎不走流量
- 易扩展
- 对小带宽 VPS 更友好

问题：

- Web 端可能受跨域影响
- 字幕、音轨、封面等附属资源不一定稳定
- 某些客户端对重定向链支持不一致

### 12.2 Why Proxy Fallback

优点：

- 兼容性更强
- 可统一 Header
- 可拦截并修复部分上游异常

问题：

- 占用网关带宽
- 服务器压力更高

因此首版策略：

- 默认 `redirect`
- Web 端或异常客户端回退 `proxy`

## 13. Security

建议至少具备以下保护：

- 管理 API 需要鉴权
- 播放 URL 支持固定 token 或签名
- 日志脱敏，不直接打印完整 cookie 或直链
- 配置文件中的敏感字段支持环境变量替换

可选增强：

- IP 白名单
- 临时签名 URL
- 基于 Emby 用户的映射认证

## 14. Observability

建议输出：

- Provider 请求耗时
- 直链缓存命中率
- 播放请求成功率
- 代理模式带宽统计
- 扫描任务成功率和耗时

首版即可通过结构化日志满足定位需要。

## 15. Failure Handling

### 15.1 Direct Link Expired

- 若缓存命中但已过期，立即刷新
- 若上游返回 403/401，清理缓存后重试一次

### 15.2 Provider Unavailable

- 返回明确错误码
- 记录日志
- 不影响其他 provider 工作

### 15.3 Partial Scan Failure

- 以目录粒度容错
- 单个 provider 或单条路径失败不应中断整个任务

## 16. Recommended MVP Scope

第一阶段只做：

1. 配置加载
2. Provider 抽象
3. 至少一个可工作的 provider
4. 全量扫描并生成 `.strm`
5. `/stream` 接口支持 `redirect` 和 `proxy`
6. SQLite 直链缓存
7. 简单任务与日志接口

不在 MVP 中实现：

- 完整 Web UI
- 增量监听
- 自动字幕同步
- 多用户权限系统
- 智能去重和电影版本合并

## 19. Final Recommendation

最适合本项目的首版路线是：

- 技术栈使用 Go
- 先做 `多 Provider + STRM 生成 + 统一播放网关`
- 默认 302，按客户端回退代理
- 让 Emby 只对接本服务生成的目录

这样可以用最小成本先跑通 115，并为后续 123pan、OpenList 和其他来源留出稳定扩展点。

## 20. Web Admin Design

### 20.1 Goal

提供一个内置 Web 管理界面，覆盖日常配置和运维场景，减少手工编辑配置文件的成本。

Web 界面主要负责：

- 管理 provider
- 管理 library 和挂载关系
- 配置播放策略
- 启动扫描任务
- 查看运行状态和日志

Web 界面不负责：

- 替代 Emby 的媒体浏览体验
- 播放器功能
- 复杂多用户协作系统

### 20.2 Design Principles

- 首版以实用为主，不追求复杂 UI
- 优先做结构化表单，不暴露原始 YAML 给用户直接编辑
- 所有页面都基于后端 API 驱动
- 页面操作结果应即时落库，并提供校验反馈
- 支持导入导出配置，方便迁移和备份

### 20.3 Frontend Scope

首版建议只做以下页面：

1. Dashboard
2. Providers
3. Libraries
4. Tasks
5. Settings

这些页面已经足够支撑 MVP 的实际使用。

### 20.4 Page Design

#### Dashboard

展示全局状态摘要：

- Provider 总数和在线状态
- 最近一次扫描时间
- 最近任务状态
- 最近播放请求结果
- 当前默认播放模式

适合展示的卡片：

- `Providers: 3 / 4 healthy`
- `Last Full Scan: 2026-04-22 01:30`
- `Playback Mode: redirect`
- `Recent Errors: 2`

#### Providers

用于管理所有数据源。

列表字段建议：

- `id`
- `type`
- `status`
- `root`
- `last_check_at`

支持操作：

- 新增 provider
- 编辑 provider
- 删除 provider
- 测试连接
- 刷新认证状态

不同 provider 的表单字段按类型动态渲染：

- `115`: cookie, root
- `123pan`: token, root
- `openlist`: base_url, access_token, root

#### Libraries

用于管理 Emby 可见的逻辑媒体库。

列表字段建议：

- `id`
- `name`
- `mount_count`
- `last_scan_at`

支持操作：

- 新增 library
- 编辑 library
- 删除 library
- 为 library 添加多个 mount
- 预览生成后的 STRM 输出路径
- 单独扫描该 library

挂载编辑表单建议包含：

- `provider`
- `source_path`
- `target_path`
- `media_type`
- `enabled`

#### Tasks

用于观察扫描和后台任务。

列表字段建议：

- `task_id`
- `type`
- `library_id`
- `status`
- `started_at`
- `finished_at`

支持操作：

- 启动全量扫描
- 启动单库扫描
- 查看任务详情
- 查看失败原因

#### Settings

用于管理全局参数。

建议包含：

- 服务监听地址
- 公网访问地址
- 默认播放模式
- Web 客户端强制代理规则
- 播放 token
- 日志等级
- 数据目录位置

### 20.5 UX Rules

界面交互建议：

- 表单提交前先做本地校验
- 提交后显示后端返回的字段级错误
- 删除操作需要二次确认
- 对敏感字段默认脱敏展示
- 对长任务使用轮询更新状态

对于 provider 测试连接：

- 应返回明确的成功/失败信息
- 失败信息要能区分认证失败、网络失败、API 失败

- `GET /api/logs/system`

### 20.7 Config Persistence Strategy

Web 管理下不建议以“原始 YAML 为唯一真相源”。

建议策略：

- 结构化配置主存储放在 SQLite
- 启动时从数据库读取配置
- 提供导出 YAML 功能作为备份和迁移格式
- 支持导入 YAML 并转成结构化配置

这样做的原因：

- 更容易做字段校验
- 更容易做版本升级
- 更适合 Web 表单增删改查
- 避免用户误改 YAML 导致服务无法启动

### 20.8 Auth Strategy

Web 管理界面至少需要基础鉴权。

首版可选方案：

1. 单用户本地管理密码
2. Basic Auth
3. Session Cookie

最小可行实现建议：

- 初始化时设置管理员账号或密码
- 登录后获取 session
- API 对管理路由校验 session

播放接口和管理接口应分离：

- `/stream/*` 使用播放 token 或签名
- `/api/*` 使用管理登录态

### 20.9 Frontend Technical Recommendation

推荐实现：

- 后端：Go
- 前端：React
- 交付方式：前端构建产物嵌入 Go 二进制

原因：

- 单文件部署简单
- 便于做内网工具型产品
- 前后端边界清晰

如果需要进一步压缩开发成本，也可以采用：

- Go 模板 + 少量原生 JS

但当页面交互变多时，维护成本会逐步升高。

### 20.11 MVP Web Scope

首版 Web 只需要覆盖以下闭环：

1. 新增一个 provider
2. 测试 provider 连接
3. 创建一个 library 和 mount
4. 触发扫描生成 `.strm`
5. 查看任务状态
6. 调整默认播放模式

做到这一步，已经能覆盖项目最核心的实际需求。
